# Temario — Git y GitHub

**Perfil de entrada:** estudiante que ya sabe algo de programación (idealmente JS Trainee o similar). No necesita experiencia previa con Git.
**Objetivo de salida:** usar Git y GitHub con fluidez para el trabajo individual y colaborativo: commits prolijos, branches, PRs, y vocabulario técnico para entrevistas.
**Conocimiento asumido por módulo:** solo los módulos anteriores de esta lista.
**Nota:** este skill no depende del lenguaje de programación — cualquier alumno puede tomarlo independientemente de si aprende JS, Python u otro lenguaje.

| # | Módulo | Subtemas | Proyecto desafío |
|---|--------|----------|-----------------|
| 01 | instalacion-y-configuracion | Qué es Git y para qué sirve (control de versiones, historia, por qué lo usa toda la industria); diferencia entre Git (herramienta local) y GitHub (plataforma remota); instalar Git (Windows: instalador .exe de git-scm.com; macOS: `xcode-select --install` o Homebrew; Linux: `apt install git` o equivalente); verificar instalación con `git --version`; configuración inicial: `git config --global user.name "Tu Nombre"`, `git config --global user.email "tu@email.com"`, configurar editor (p. ej. VS Code con `git config --global core.editor "code --wait"`); ver config con `git config --list`; crear cuenta GitHub (github.com); autenticación — SSH keys: generar con `ssh-keygen -t ed25519 -C "email"`, agregar a GitHub en Settings → SSH and GPG keys, verificar con `ssh -T git@github.com`; alternativa HTTPS con token personal (Settings → Developer settings → Personal access tokens); cuándo usar SSH vs HTTPS | "Mi primera configuración" — crear un repo local con un README.md que describe al alumno (nombre, ciudad, por qué estudia programación), subirlo a GitHub con SSH o HTTPS |
| 02 | fundamentos-de-git | `git init` (qué crea la carpeta oculta `.git` y por qué existe); `git status` (entender los tres estados: untracked, staged, modified, committed); `git add` (staging area — qué es, por qué existe, la analogía de la "mesa de preparación"); `git add .` vs `git add archivo`; `git commit -m "mensaje"` (anatomía de un commit: hash SHA-1, autor, fecha, mensaje, snapshot); buenas prácticas de mensajes de commit (presente imperativo, conciso, descriptivo); `git log` (con flags `--oneline`, `--graph`, `--all`); `git show <hash>`; `.gitignore` (qué ignorar y por qué: `node_modules/`, `.env`, archivos de OS como `.DS_Store` y `Thumbs.db`, archivos de IDE); diferencia conceptual entre working directory / staging area / repository (los "tres árboles" de Git) | "Historial de una receta" — crear un repo que registre la evolución de un archivo de texto (una receta que se va mejorando) con al menos 5 commits bien descritos, cada uno representando una versión distinta |
| 03 | trabajar-con-github-remoto | `git remote add origin <url>`; `git push -u origin main` (y qué hace el flag `-u`); `git pull`; `git clone <url>`; diferencia entre `git fetch` y `git pull` (fetch descarga sin fusionar; pull = fetch + merge); GitHub interface: explorar un repositorio (commits, ramas, archivos, historial de cambios); README.md con Markdown (encabezados, listas, código, imágenes, badges); GitHub Pages: activar en Settings → Pages, elegir rama `main` y carpeta `/root`, deploy básico de un sitio HTML/CSS estático; entender las URLs de GitHub Pages (`usuario.github.io/repo`) | "Mi portfolio en GitHub Pages" — subir un mini-sitio HTML/CSS (puede ser el proyecto de HTML/CSS del bootcamp, o uno nuevo) a GitHub Pages y compartir la URL pública |
| 04 | branching-y-merging | Qué es una branch y por qué existe (trabajo aislado para features, bugfixes, experimentos sin romper `main`); `git branch` (listar, crear); `git checkout -b <nombre>` y la forma moderna `git switch -c <nombre>`; `git checkout <nombre>` / `git switch <nombre>` para cambiar de rama; `git merge <rama>` (estando en la rama destino); Fast-forward merge vs 3-way merge (cuándo ocurre cada uno y qué diferencia hace); merge conflicts: cómo se ven en el archivo (marcadores `<<<<<<`, `=======`, `>>>>>>>`), cómo resolverlos manualmente y completar el merge con `git add` + `git commit`; `git branch -d <nombre>` para eliminar ramas ya mergeadas; `git log --graph --oneline --all` para visualizar el árbol de commits; convenciones de nombres de ramas: `feature/nombre-feature`, `fix/descripcion-bug`, `hotfix/descripcion` | "App con dos features simultáneas" — partiendo de un repo base (un archivo HTML o JS simple), crear dos ramas (`feature/dark-mode` y `feature/contador`), desarrollar ambas independientemente con al menos 2 commits cada una, y mergearlas a `main` resolviendo el conflicto que inevitablemente aparece |
| 05 | flujo-colaborativo | Fork (qué es y cuándo se usa: contribuir a repos ajenos) vs clone (trabajar en repos propios o del equipo); crear un Pull Request en GitHub (desde la interfaz): título, descripción (qué cambia, por qué, cómo probarlo), asignar reviewers, agregar labels; code review básico: cómo dejar comentarios útiles (específicos, sobre el código no la persona, con sugerencias concretas), aprobar o solicitar cambios; Issues en GitHub: abrir un issue (título, descripción, labels), asignar a un colaborador, cerrar un issue desde un commit con la palabra clave `Closes #N`; GitHub Flow completo: una rama por feature → PR → review → merge → (deploy); Conventional Commits: `feat: descripción`, `fix: descripción`, `chore: descripción`, `docs: descripción`, `refactor: descripción`, `test: descripción` — por qué esta convención es útil para equipos y herramientas automatizadas; cómo trabaja un equipo real: asignación de tareas con Issues, ramas por tarea, PRs para revisión, merge solo después del review | "Simulación de team workflow" — el alumno hace fork de un repo provisto por el instructor (o crea uno y agrega un colaborador), trabaja en una feature en una rama con al menos 3 commits con Conventional Commits, y crea una PR con título y descripción completos (qué cambia, cómo probarlo) |
| 06 | comandos-intermedios | `git stash` (guardar trabajo en progreso temporalmente cuando hay que cambiar de contexto); `git stash pop` y `git stash list`; `git revert <hash>` (crear un nuevo commit que deshace uno anterior — forma segura en repos compartidos); `git reset` modos: `--soft` (mueve HEAD, mantiene staged), `--mixed` (mueve HEAD, unstaged), `--hard` (mueve HEAD, descarta cambios — PELIGRO en repos compartidos); diferencia clave: `revert` es seguro en equipos, `reset --hard` reescribe la historia y puede causar caos si ya hiciste push; tags: `git tag v1.0.0`, `git tag -a v1.0.0 -m "mensaje"` (anotados), `git push origin v1.0.0`, `git tag` para listar; `git log` avanzado: `--author`, `--since`, `--until`, `--grep`; `git diff` (diferencia en working directory vs staged); `git cherry-pick <hash>` (mención: aplicar un commit específico a otra rama, cuándo usarlo); `git bisect` (mención: encontrar el commit que introdujo un bug por búsqueda binaria) | "Historial limpio" — a partir de un repo con historial desordenado provisto por el instructor (con commits mal descritos, cambios mezclados, cosas a deshacer), aplicar revert para deshacer un commit problemático, usar stash para cambiar de contexto, y crear un tag de versión |
| 07 | buenas-practicas-y-proyecto-integrador | Mensajes de commit que comunican: Conventional Commits en profundidad, ejemplos de mensajes malos vs buenos, cómo el historial prolijo ayuda al equipo y a las herramientas de CI/CD; `.gitignore` por tipo de proyecto: plantillas para Node.js, Python, .NET, proyectos generales (referencia a gitignore.io); README profesional: descripción del proyecto, badges de estado, instrucciones de instalación, instrucciones de uso, cómo contribuir, licencia — el README es la "carta de presentación" del proyecto; GitHub profile README (el repo especial `usuario/usuario`); git hooks (mención: qué son, pre-commit hooks para correr linters antes de commitear, herramienta Husky para Node.js); estrategias de branching comparadas: Git Flow (ramas develop, release, hotfix — para releases con versiones), GitHub Flow (solo main + feature branches — para deploy continuo), Trunk-Based Development (todos en main con feature flags — para equipos avanzados); cuándo usar cada estrategia; cómo prepararse para una entrevista que incluye Git: conceptos que siempre preguntan, cómo demostrar fluencia | Proyecto integrador: subir un proyecto real propio (puede ser el proyecto integrador de JS Trainee u otro) con historial de commits prolijo (mínimo 8 commits con Conventional Commits), README profesional con todas las secciones, `.gitignore` correcto para el tipo de proyecto, y al menos una PR simulada (crear una rama, hacer cambios, abrir la PR, mergearla con descripción completa) |

## Notas de progresión

- **El skill no es JS-dependent:** cualquier alumno puede tomarlo independientemente del lenguaje que aprende, siempre que tenga alguna experiencia programando.
- **Módulos 01-04 son secuenciales** y deben completarse en orden: no tiene sentido hablar de merge conflicts antes de entender qué es un commit.
- **El módulo 05 asume** que el alumno tiene cuenta de GitHub activa y sabe hacer push/pull cómodamente.
- **En el módulo 06, enfatizar fuertemente el peligro de `git reset --hard` en repos compartidos.** Es uno de los errores más costosos en equipos reales: reescribe la historia del repositorio, y si ya hiciste push, obliga a todos los colaboradores a hacer `git pull --force` o perder trabajo. La regla de oro: `revert` para deshacer en repos compartidos; `reset` solo en ramas locales que todavía no publicaste.
- **El módulo 07 es integrador:** no introduce comandos nuevos complejos, sino que consolida todo en el contexto de prácticas profesionales. Es el módulo más importante para la empleabilidad.
